# Работа с HTTP Cookie

Подсистема работы с cookie во фреймворке **Joke** предоставляет типобезопасный, удобный и безопасный по умолчанию
механизм управления состоянием клиента. Она абстрагирует работу с заголовками `Set-Cookie` и `Cookie`, автоматически
решая задачи кодирования данных, установки сроков жизни и применения политик безопасности.

## Основные принципы

1. **Безопасность по умолчанию (Secure by Default).** Все создаваемые куки по умолчанию имеют флаги `HttpOnly` и
   `Secure`, а также политику `SameSite=Lax`. Это защищает приложение от XSS и CSRF атак без дополнительных настроек.
2. **Иммутабельность конфигурации.** Настройки cookie замораживаются при загрузке приложения, что гарантирует
   предсказуемое поведение в течение всего запроса.
3. **Автоматическое кодирование.** Специальные символы в значениях cookie автоматически кодируются при записи и
   декодируются при чтении.
4. **Уникальность по контексту.** Поддерживается хранение нескольких cookie с одинаковым именем для разных доменов или
   путей.

## Конфигурация

Глобальные настройки cookie определяются в классе `CookieConfig`. Они применяются ко всем cookie, создаваемым через
коллекцию ответа, если параметры не переопределены явно.

По умолчанию используются следующие значения:

| Параметр   | Значение по умолчанию | Описание                                                 |
|:-----------|:----------------------|:---------------------------------------------------------|
| `lifetime` | `31536000` (1 год)    | Время жизни в секундах. `null` означает сессионную куку. |
| `path`     | `/`                   | Доступна для всех путей домена.                          |
| `domain`   | `null`                | Привязка только к текущему хосту (Host-only).            |
| `secure`   | `true`                | **Только HTTPS.**                                        |
| `httpOnly` | `true`                | Запрет доступа из JavaScript.                            |
| `sameSite` | `Lax`                 | Защита от CSRF с сохранением удобства навигации.         |

### Важное предупреждение для локальной разработки

Поскольку по умолчанию установлен флаг `secure => true`, браузеры **отвергнут** такие cookie при работе по протоколу
HTTP (например, на `http://localhost`).

Для локальной разработки необходимо явно отключить этот флаг в конфигурационном файле приложения (обычно в файле `config/app.php`):

```php
use Vasoft\Joke\Http\Cookies\CookieConfig;

// ... внутри конфигурационного файла или провайдера

$cookieConfig = new CookieConfig();
$cookieConfig->setSecure(false);

return [
    CookieConfig::class => $cookieConfig
];
```

> **Совет:** В продакшене всегда оставляйте `setSecure(true)`. Фреймворк автоматически подстроится под HTTPS соединение.

## Установка Cookie

Для добавления cookie используется объект `CookieCollection`, доступный через объект ответа (`Response`).

### Базовое использование

Достаточно передать имя и значение. Остальные параметры (время жизни, флаги) будут взяты из глобальной конфигурации.

```php
use Vasoft\Joke\Http\Response\HtmlResponse;

// Внутри контроллера или middleware
public function handle(HtmlResponse $response): HtmlResponse
{
    // Установит куку с настройками из CookieConfig
    $response->cookies->add('theme', 'dark');
    
    return $response->setBody('Настройка темы сохранена');
}
```

### Переопределение параметров

Вы можете переопределить любые параметры для конкретной куки, передав их именованными аргументами:

```php
$response->cookies->add(
    name: 'session_token',
    value: 'abc-123-xyz',
    lifetime: 3600,           // Живет 1 час
    path: '/admin',           // Доступна только в /admin
    httpOnly: true,           // Явное включение защиты
    sameSite: \Vasoft\Joke\Http\Cookies\SameSiteOption::Strict
);
```

### Кодирование данных

Вам не нужно беспокоиться о специальных символах. Фреймворк автоматически выполнит `rawurlencode` для значения:

```php
// Значение "Привет, мир!" будет корректно закодировано в заголовке
$response->cookies->add('greeting', 'Привет, мир!'); 
```

## Чтение Cookie

Входящие cookie доступны через объект запроса `HttpRequest` в свойстве `$cookies`. Значения автоматически декодируются 
(`urldecode`), поэтому вы работаете с исходными строками.

```php
use Vasoft\Joke\Http\HttpRequest;
use Vasoft\Joke\Http\Response\HtmlResponse;

public function showProfile(HttpRequest $request, HtmlResponse $response): HtmlResponse
{
    // Получение значения (автоматически декодировано)
    $theme = $request->cookies->get('theme', 'light'); // 'light' - значение по умолчанию
    
    // Проверка существования
    if ($request->cookies->has('session_id')) {
        $sessionId = $request->cookies->get('session_id');
        // ... логика авторизации
    }

    return $response->setBody("Ваша тема: {$theme}");
}
```

> **Примечание:** Класс `InputCookieCollection` наследует методы коллекции свойств (`get`, `has`, `all`), предоставляя
> удобный API для работы с данными.

## Удаление Cookie

Для удаления cookie необходимо отправить браузеру инструкцию с тем же именем, доменом и путем, но с истекшим временем
жизни. Используйте метод `remove()`.

```php
public function logout(HttpRequest $request, HtmlResponse $response): HtmlResponse
{
    // Удаляет куку, используя параметры по умолчанию из конфига
    $response->cookies->remove('session_token');
    
    return $response->setBody('Вы вышли из системы');
}
```

### Точность удаления

Если кука была создана с нестандартными параметрами (например, для конкретного домена или пути), эти параметры **должны
совпадать** при удалении.

```php
// Если кука была создана так:
$response->cookies->add('admin_panel', '1', domain: 'admin.example.com', path: '/dashboard');

// То удалить её нужно так:
$response->cookies->remove('admin_panel', 'admin.example.com', '/dashboard');
```

Если вы не укажете домен и путь в методе `remove()`, фреймворк попытается использовать значения из текущей конфигурации.
Это сработает корректно, только если конфиг не менялся с момента создания куки.

## Продвинутые сценарии

### Мульти-доменные Cookie

Фреймворк поддерживает установку нескольких cookie с одинаковым именем для разных доменов или путей. Они не перезапишут
друг друга.

```php
// Кука для основного домена
$response->cookies->add('lang', 'ru', domain: 'example.com');

// Кука для поддомена (не затронет основную)
$response->cookies->add('lang', 'en', domain: 'en.example.com');
```

### Сессионные Cookie

Чтобы создать куку, которая живет только до закрытия браузера, установите `lifetime` в `null`:

```php
$response->cookies->add('temp_data', 'value', lifetime: null);
```

### Политики SameSite

Используйте enum `SameSiteOption` для управления отправкой cookie при кросс-сайтовых запросах:

* `SameSiteOption::Strict` — Максимальная защита. Кука не отправляется при переходах с внешних сайтов.
* `SameSiteOption::Lax` (по умолчанию) — Безопасный баланс. Кука отправляется при GET-запросах (переход по ссылке), но
  не при POST.
* `SameSiteOption::None` — Кука отправляется везде. **Требует обязательного наличия флага `Secure` (HTTPS).** Фреймворк
  автоматически добавит флаг `Secure`, если вы выберете этот режим, даже если в конфиге он выключен.

```php
use Vasoft\Joke\Http\Cookies\SameSiteOption;

$response->cookies->add(
    'tracking_id', 
    'xyz', 
    sameSite: SameSiteOption::None
    // Флаг Secure будет добавлен автоматически
);
```

## Архитектурные ограничения (v1.x)

В текущей версии фреймворка (1.x) определение флага `Secure` зависит от глобальной конфигурации. Автоматическое
переключение режима (HTTP → `Secure=false`, HTTPS → `Secure=true`) на уровне коллекции пока не реализовано, чтобы
сохранить обратную совместимость конструкторов.

**Рекомендация:** Используйте переменные окружения для управления конфигом:

* В `.env.production`: `COOKIE_SECURE=true`
* В `.env.local`: `COOKIE_SECURE=false`

В версии 2.0 планируется внедрение автоматического определения протокола запроса для динамического управления флагом
`Secure`.
